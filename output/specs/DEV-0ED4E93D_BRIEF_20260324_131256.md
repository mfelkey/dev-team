## 2. Rate Limiting Architecture

- **Tiered System**: 
    - **Free Tier**:
        - GET, POST: 100 requests/hour
    - **Basic Tier**:
        - GET, POST: 500 requests/hour
    - **Pro Tier**:
        - GET, POST: Unlimited (or very high threshold)

- **Implementation**: Use Redis to track rate limits for each user by subscription tier.

## 3. Authentication Architecture

- **JWT vs Session**: JWT is recommended due to its stateless nature and ease of scaling.
- **Token Refresh Strategy**: Implement token refresh with short-lived access tokens (~15 minutes) and long-lived refresh tokens (7 days).
    - Access Token:
        ```json
        {
            "exp": 1689430297,
            "user_id": "uuid",
            "subscription_tier": "string"
        }
        ```
    - Refresh Token: Stored securely on the client-side and used to fetch new access tokens.
- **Subscription Tier Enforcement**: Validate subscription tier during every authenticated request.

## 4. DS Integration Contract

### Data Quality Handling:
- BLOCK (<0.70): Do not serve data.
- FLAG (0.70-0.85): Serve data but flag in response.
- PASS (>0.85): Serve data normally.

- **Polling vs Push**: Use a combination of both for real-time and scheduled updates.
    - Poll for periodic checks, push when new matches or significant model updates occur.

### Caching Strategy:
- Cache analytics results for 1 hour using Redis with expiration.
- Invalidate cache on new match data or updated model predictions.

## 5. Database Access Pattern

### Match Data Endpoints:
- Direct DB access for:
    - GET `/api/matches` (if no cached entry)
- Read from cache first, then read directly from the DB if not found in Redis.
- **Connection Pooling**: Use connection pooling with `psycopg2` to manage PostgreSQL connections efficiently.
