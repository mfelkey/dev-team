# ParallaxEdge Railway Infrastructure Spec (ID: DEV-B395578D)

## 1. RAILWAY PROJECT STRUCTURE

The following services will be provisioned on Railway and their relationships are detailed below:

- **Next.js Frontend** — Deploys the frontend application.
- **FastAPI Backend** — Hosts API endpoints for serving model outputs and handling user requests.
- **PostgreSQL Database** — Stores pre-computed model outputs.
- **Redis Cache/Rate Limiter** — Handles rate limiting and caching of frequent read queries.
- **Python Workers (Background Jobs)** — Executes scheduled tasks such as weather polling, odds polling, and triggering model inference.

### Service Provisioning

1. **Next.js Frontend**
   - Name: `frontend`
   - Connects to: GitHub repository (`mfelkey/parallaxedge`), deploys from `/client`

2. **FastAPI Backend**
   - Name: `backend`
   - Connects to: PostgreSQL database, Redis cache/rate limiter

3. **PostgreSQL Database**
   - Name: `database`
   - Serves data to the backend service.

4. **Redis Cache/Rate Limiter**
   - Name: `redis-cache`
   - Configured for rate limiting and caching tasks

5. **Python Workers (Background Jobs)**
   - Name: `workers`
   - Connects to PostgreSQL, Redis

### Relationships

- The frontend connects directly to the backend API.
- The backend interacts with both the PostgreSQL database and Redis cache/rate limiter.
- Python workers update the database and cache as needed.

## 2. ENVIRONMENT VARIABLES

- **Next.js Frontend**
  - `NEXT_PUBLIC_API_URL` — URL of FastAPI backend (e.g., `https://backend.parallaxedge.com`)
  
- **FastAPI Backend**
  - `DATABASE_URL` — PostgreSQL connection string
  - `REDIS_URL` — Redis connection string for cache and rate limiting
  - `WORKER_POLLING_INTERVAL_WEATHER` — Interval for weather polling in seconds (e.g., "1800")
  - `WORKER_POLLING_INTERVAL_ODDS` — Interval for odds polling in minutes (e.g., "5")
  - `MODEL_INFERENCE_TRIGGER_TIME` — Time to trigger model inference daily (e.g., "23:00")

- **PostgreSQL Database**
  - No specific environment variables, managed through Railway service configuration.

- **Redis Cache/Rate Limiter**
  - No specific environment variables, managed through Railway service configuration.

## 3. POSTGRESQL SCHEMA DEPLOYMENT

### Migration Tool
Use Alembic for database migrations.

### Deployment Steps on Railway:
1. Configure the `railway.toml` with the PostgreSQL connection string.
2. Run `alembic revision --autogenerate -m "Initial schema"` to create a migration file.
3. Apply migrations using `alembic upgrade head`.

### Initial Setup
- Use SQLAlchemy for ORM in FastAPI.
- Models should include tables for storing model outputs (`model_outputs`), inputs snapshot, and flags.

## 4. REDIS CONFIGURATION

### Railway Redis Plan
Choose the free tier or a suitable paid plan based on expected cache size and traffic.

### Key Naming Conventions
- **Rate Limiting**: `rl:<user_id>:<resource>`
- **Caching**: `cache:<model_output_key>`

### TTL Recommendations
- Rate limiting keys: 1 hour (3600 seconds)
- Caches for model outputs: 24 hours (86400 seconds)

## 6. BACKGROUND WORKER SETUP

Scheduled jobs will be handled using Railway's Cron service.

### Cron Syntax Examples:

1. Weather Polling (pre-game intervals)
   ```bash
   0 */3 * * * python worker/weather_poller.py
   ```

2. Odds Polling (every 5 min on match day)
   ```bash
   */5 * * * * python worker/odds_poller.py
   ```

3. Model Inference Trigger (daily at midnight)
   ```bash
   0 0 * * * python worker/inference_trigger.py
   ```

## 9. FIRST DEPLOY CHECKLIST

1. **Install Railway CLI**: `npm install -g @railway/cli`
2. **Login to Railway**: `railway login`
3. **Create a new project**: `railway init --name parallaxedge`
4. **Deploy frontend**:
   ```bash
   cd client
   railway deploy --env NEXT_PUBLIC_API_URL=https://backend.parallaxedge.com
   ```
5. **Deploy backend and services**:
   ```bash
   cd server
   railway services add postgresql database
   railway services add redis redis-cache
   railway deploy
   ```
